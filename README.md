# SherdFusion

SherdFusion is a staged pipeline for pottery sherd matching. The repository is organized around three main learning stages and several dataset utilities:

- Stage 1: fragment retrieval
- Stage 2: pose estimation by diffusion-based pair alignment
- Stage 3: pairwise compatibility classification

The project also includes compact dataset builders and a gold-standard evaluation branch for end-to-end verification.

## Repository Layout

- `dataset_simulated_sample_1000/`
  Builds the compact simulated pair dataset used by Stage 1 and Stage 2.
- `dataset_builder/`
  Utilities for preparing additional datasets.
- `dataset_gold_standard/`
  Gold-standard benchmark assets.
- `stage_1_retrieval_network/`
  Learns fragment embeddings and evaluates retrieval metrics such as `R@5`, `R@10`, `NDCG@5`, and `NDCG@10`.
- `stage_2_pose_estimation/`
  Trains and evaluates a diffusion model for pairwise pose alignment.
- `stage_3_pairwise_compatibility/`
  Trains a binary classifier to decide whether an exterior/interior pair is compatible.
- `gold_standard_end_to_end/`
  End-to-end evaluation scripts for the curated benchmark.

## Overall Pipeline

1. Build the simulated compact pair dataset.
   Input: grouped simulated fragments.
   Output: `simulated_dataset_pairs.pkl` containing unique fragments, pair indices, and images.

2. Train the retrieval network.
   Stage 1 learns fragment embeddings from the compact simulated dataset.
   Typical outputs: retrieval checkpoints, retrieval metrics CSV, and top-k visualizations.

3. Train the pose estimation model.
   Stage 2 uses the same compact pair dataset and learns pairwise rigid alignment with a diffusion model.
   Typical outputs: alignment checkpoints, validation metrics, and train/validation visualizations.

4. Evaluate pose estimation on the validation split.
   The Stage 2 test script reconstructs the validation split and reports recall-style pose metrics such as `AEtrans`, `AErot`, and `Avg. IoU`.

5. Train the pairwise compatibility classifier.
   Stage 3 consumes labeled exterior/interior image pairs and predicts whether a candidate pair is compatible.
   Typical outputs: `best_model.pt`, split CSV files, training curves, and classification history.

6. Run end-to-end or benchmark evaluation.
   The gold-standard branch combines retrieval and pose verification for curated benchmark pairs.

## Data Formats

### Compact Simulated Pair Dataset

The compact simulated dataset generated by `dataset_simulated_sample_1000/build_paired_dataset.py` is stored as six sequential pickle dumps:

1. `all_polys`
2. `all_local_features`
3. `all_global_features`
4. `all_actions`
5. `positive_pair_indices`
6. `all_fragment_images`

This layout keeps each fragment only once and stores positive pairs as index tuples, which is much smaller than duplicating full pair records.

### Pairwise Compatibility Labels

Stage 3 expects CSV files with these columns:

- `group_name`
- `label`
- `exterior_image`
- `interior_image`
- `exterior_path`
- `interior_path`

## Main Entry Points

- Stage 1 training and evaluation:
  `stage_1_retrieval_network/search.py`
- Stage 2 training:
  `stage_2_pose_estimation/train/train_pair.py`
- Stage 2 testing:
  `stage_2_pose_estimation/test/test.py`
- Stage 3 label merge:
  `stage_3_pairwise_compatibility/merge_labels.py`
- Stage 3 training:
  `stage_3_pairwise_compatibility/train.py`
- Stage 3 inference:
  `stage_3_pairwise_compatibility/infer.py`

## Typical Workflow

1. Generate or locate `simulated_dataset_pairs.pkl`.
2. Train Stage 1 retrieval on the simulated dataset.
3. Train Stage 2 pose estimation on the same compact pair dataset.
4. Evaluate Stage 2 using the saved validation split configuration.
5. Merge positive and negative CSV labels for Stage 3.
6. Train the Stage 3 compatibility classifier.
7. Run benchmark scripts on curated gold-standard data when needed.

## Notes

- Several scripts assume local research-style paths and are not packaged as a library.
- Checkpoint files may store training arguments in addition to weights.
- For large datasets, image loading is intentionally delayed in some scripts to reduce startup cost.
